# 🔧 全局配置 (所有 Skill 共享)

## 根目录
所有 Skill 中的相对路径（如 `01-Clients/...`）均基于以下根目录解析：
```
ROOT: ./Obsidian-Vault
```

## 系统标准目录树 (全局向导)
当需要在系统中创建新文件或目录时，**必须严格遵守以下目录架构**：
```
ROOT/
├── 00-RawData/       # [前端/原始] 各类脱敏前的原始音视频、问卷数据
├── 01-Clients/       # [个案管理] 按儿童代号拆分，存储核心档案、评估、FBA、IEP、转衔
├── 02-Sessions/      # [日常记录] 按儿童代号拆分，存储课后记录、行为 ABC 日志
├── 03-Staff/         # [师资管理] 教师成长档案、听课反馈、下发给老师的具体实操单
├── 04-Supervision/   # [公共基建] 督导个人的复盘、灵感库、系统变更日志
├── 05-Communication/ # [对外沟通] 面向家长的阶段报告、周报家书、战前简报
└── 06-Templates/     # [模板库] 督导要求一线人员填写的各类空白表单模板
```
任何未被定义在此目录树中的顶级文件夹都视为违规建立。

## 占位符解析规则
- **`[代号]`**：完整格式为 `A-兜兜`，用于路径时为 `Client-A-兜兜`
  - 如用户只说"兜兜"，先列出 `{{ROOT}}/01-Clients/` 下所有目录，匹配含"兜兜"的目录
  - 如有多个匹配，列出候选请用户确认
- **`[教师姓名]`**：用于路径时为 `教师-[姓名]`，如 `教师-小李`
  - 如用户只说"小李"，先列出 `{{ROOT}}/03-Staff/` 下所有目录，匹配含"小李"的目录
- **解析失败时**：列出已有目录，请用户确认。绝不猜测。

## 日期格式规范
- **`{{当前日期}}`** → `YYYY-MM-DD` 格式，如 `2026-03-06`
- **`{{当前日期时间}}`** → `YYYY-MM-DD HH:mm` 格式，如 `2026-03-06 19:00`
- **文件名中的日期** → 一律使用 `YYYY-MM-DD`
- **正文/家书中** → 可用更友好的格式：`YYYY年M月D日`
- 注意：`{{date}}` 和 `{{日期}}` 均等价于 `{{当前日期}}`

## 报告类文件名日期铁律（OBS-21 防护）

所有"报告类"/"周期类"文件名一律用 **执行日期 `{{当前日期}}`**，**不是事件日期**：

### ✅ 正确
- `家书-2026-04-18.md`（本信谈 2026-04-28 的课程，但文件在 2026-04-18 生成）
- `喜报-里程碑-2026-04-18.md`
- `Client-A-小哲-里程碑报告-2026-04-18.md`
- `Client-A-小哲-强化物评估-2026-04-18.md`
- `复盘-2026-04-18.md`
- `2026-04-28-Client-A-小哲-Teacher A记录.md`（课后记录例外 — 文件名日期 = 上课日）

### ❌ 错误（会导致同一孩子多次跑同一 skill 时覆盖）
- `家书-2026-04-28.md`（parent_letter 用事件日期作文件名 — 下次跑若事件也是 04-28 会覆盖）

### frontmatter 记录事件日期
正文涉及的事件区间通过 frontmatter 记录（OBS-02 规范）：

```yaml
---
date: 2026-04-18              # 档案生成日（执行日期）
数据窗口截止: 2026-04-28       # 材料引用的最晚事件日
---
```

**例外**（仅以下 skill 文件名日期 = 事件日，因文件名必须和课程一一对应）：
- `session-reviewer` 的课后记录：`{{课程日期}}-Client-[代号]-[教师]记录.md`

## 预览确认标准流程
当 Skill 安全协议要求"预览确认"时，Claude 必须遵循以下流程：
1. 用 `> [!NOTE] 📋 变更预览` callout 包裹预览内容
2. 如果是 **Edit 操作**（替换已有章节），用并排对比格式：
   - `**原内容：**` + 原文
   - `**拟替换为：**` + 新文
3. 如果是 **Write 操作**（新建文件），直接展示全文
4. 如果预览内容超过 50 行，先给出摘要（核心变更点），询问是否查看完整内容
5. 预览后必须明确询问：**"以上变更是否确认执行？(确认/修改/取消)"**
6. 只有收到明确肯定回复（如"确认"、"好"、"y"、"执行"）后才执行写入

## 文件不存在时的处理策略
当 Skill 的 Read 步骤中需要读取的文件不存在或为空占位文件时：
1. **不要报错终止**：基于已有数据继续执行 Skill
2. **标记为"待完善"**：在输出中用 `⏳ [待XX完成后补充]` 占位，绝不编造数据
3. **执行完成后提醒**：汇报中注明"建议先执行 [缺失的skill名] 来补全数据"

**⚠️ 特殊情况：公共基建文件的 Append 容错**
当向 `04-Supervision/系统变更日志.md` 或 `04-Supervision/督导灵感与SOP迭代库.md` 追加内容时，**如果该文件不存在，Claude 必须主动进行无中生有（Create）**：
1. 建立具有对应 `# 标题` 的 Markdown 框架。
2. 再执行追加写入。
**绝对不允许因为“找不到变更日志文件”而中断 Skill 执行。**

## SECTION_REPLACE / EDIT 操作内容边界（BUG #21 变种防护）

当 skill 使用 `| EDIT: <章节名>` 或 `| SECTION_REPLACE: <章节名>` 修饰符编辑已有档案章节时，你提供的**内容**就是 **章节正文**（不含章节标题本身）。vault_service 会自动保留原章节标题，只替换其下方的正文。

### 规则
- ✅ 你只输出**章节正文**（不含 `## xxx` 标题）
- ❌ 不要在内容首行重复该章节的 `## xxx` 二级标题
- ❌ 不要给该章节换一个新标题（如果确实需要改标题，新开一个 skill 或走 `WRITE` 全文覆盖模式）

### 错误示例（v4 测试发现 — 产生双二级标题）
```
<!-- FILE: Client-A-小哲-核心档案.md | SECTION_REPLACE: ## ⚠️ 历史问题行为备忘 -->
## ⚠️ 问题行为预警（基于 2026-04-18 fba-analyzer 初步分析）
> ⚠️ 数据状态 ...
```

结果会在核心档案中产生：
```
## ⚠️ 历史问题行为备忘        ← 系统保留的原标题
                                ← 空行（原标题章节被清空）
## ⚠️ 问题行为预警...           ← AI 内容自带的新标题
```

### 正确示例
```
<!-- FILE: Client-A-小哲-核心档案.md | SECTION_REPLACE: ## ⚠️ 历史问题行为备忘 -->
> ⚠️ **数据状态**：本节基于 6 个 ABC Event 初步分析...

### 当前高频行为一：被动非回应型任务逃避
- **行为拓扑**：沉默 15–30s → 仿说末尾词...
```

### 允许的首行开头
- ✅ `> [!warning] 数据状态：...` callout
- ✅ `**数据状态**：...` 粗体段落
- ✅ `### 当前高频行为一：xxx`（三级标题，作子分组）
- ✅ `- 项目一：...` 列表项

## 🛑 真名在正式档案绝对禁令 (P0 — BUG #24 对应)

**适用范围**：除 `00-RawData/身份映射对照表-绝密.md` 以外的**所有**档案：
- 正文内容 ❌ 不得出现身份映射表中的任何真名
- frontmatter ❌ 不得出现真名
- **meta-commentary / 合规声明 / 脱敏说明** ❌ **亦不得引用真名**（这是最易踩坑的 AI 行为）

**BUG #24 典型错误（严禁模仿）**：

```
❌ 错误示范：
5. 原始资料中出现的父母姓名（李建国/陈婉仪）：已通过 `privacy-filter` 原则处理，
   正式档案中不留存；本报告仅以角色称谓（父亲/母亲）引用
```

上面这段话的**意图**是解释"已脱敏"，但**行为**是把真名搬进了正式档案——这是
**自毁式脱敏**。AI 在所有下游 skill 输出中**都不能**有这种 meta-commentary。

**正确写法**：

```
✅ 正确示范（如果需要说明脱敏策略）：
5. 本报告涉及的家庭成员均以角色代称（父亲/母亲/外婆）引用，真实身份信息仅保存于
   绝密映射表。
```

**根因理解**：下游 skill 在读 fixture / upload body 时会看到真名。AI 可能会
"好心"地在输出里说明"我脱敏了"，但**任何引用原始真名的 meta-commentary 都是
脱敏的失败**——正确做法是**完全不提"原始姓名是什么"**，只提"本报告使用角色代
称"。

**自检规则**（每个下游 skill 输出前必做）：
- [ ] 扫描即将写入的每个 FILE marker 下的文本
- [ ] 若文本里出现原始上传材料中的任何真名（身份映射表里的姓名），**立即删除
  整句**或改写为角色代称
- [ ] 特别检查"原始/真实/原名/真名"等元词汇附近是否夹带真名——这些是最常见的
  meta-leak 信号词
- [ ] 若发现 meta-leak，在响应中明确输出 `⚠️ 触发真名 meta-leak 自检，已重
  写段落 [X]`，不得静默通过

## 📑 事实 / 观察 / 推断 三层分层规范（OBS-03）

任何产出在**事实性档案**（能力评估、FBA 分析、核心档案、IEP）里的陈述，必须按来源分层标注。在 v2/v3/v4 三轮测试中命中频率极高（v2 "听觉敏感/触觉防御/口腔感觉寻求" 只有一项有"疑似"；v3 给班主任的三步方案以"操作指令"口吻写却未标推断层）。

### 四种层级（均使用 Obsidian callout）

| 层级 | 语法 | 适用场景 | 示例 |
|:---|:---|:---|:---|
| 事实层 | 普通段落 | 家长原话、量表原始分、可验证事件 | 母亲原话："他每天早上 6:30 起床" |
| 观察层 | `> [!observation]` | 督导/老师当场目视的行为描述 | `> [!observation] 访谈中儿童出现 2 次，每次 5 分钟，眼神接触 1 次` |
| 推断层 | `> [!hypothesis]` 或句末加 `（临床推断）` | AI 或 BCBA 从数据推出的结论 | `> [!hypothesis] 功能假设：逃避维持（Escape-maintained）` |
| 预测层 | `> [!prediction]` | 对未来 N 周的走势预测 | `> [!prediction] 若 ST-1 持续 3 周 80%+，可在 2026-05-19 进入 ST-1.2` |

### 典型错误（严禁模仿）

❌ "感官特征：听觉敏感；触觉防御；口腔感觉寻求（疑似）" — 三项并列但只有一项有"疑似"；其余两项实际也是推断

✅ 正确分层：
```
**感官特征**：
- **触觉防御**：湿黏触感厌恶（家长原话 2026-04-24 访谈）
> [!hypothesis] 听觉敏感 / 口腔感觉寻求
> 从"对洗发水味敏感"、"只吃 5 种主食"等线索推断，需观察验证。
```

### 适用 skill（必用 callout）
- `assessment-logger` / `fba-analyzer` / `profile-builder` / `plan-generator` / `milestone-report`

## 📋 统一 frontmatter 字段（OBS-17 规范）

所有事实类档案（能力评估 / FBA / IEP / 里程碑 / session 日志）使用以下统一字段：

```yaml
---
aliases: [档案名]
tags: [个案/类型, 个案/A-代号]
child_alias: A-代号
date: {{当前日期}}                    # 档案生成日（执行日期）
数据窗口截止: YYYY-MM-DD              # 最晚事件日（OBS-02 规范）
data_quality: 🟢直接观察 / 🟡报告摘录 / 🟠口述推定 / 🔴初步估算
data_sources:                          # 列表，所有数据来源文件
  - 2026-04-28 课后记录
  - 2026-04-25 ABC 日志
  - 家长口述 VB-MAPP 2026-04-24
---
```

**必填字段**：`aliases`、`tags`、`child_alias`、`date`
**数据类档案额外必填**：`数据窗口截止`、`data_quality`、`data_sources`
**家书 / 喜报 / 复盘等非数据类档案**：可省略 `data_quality` / `data_sources`

任何"推断"、"结论"、"预测"都必须能从 `data_sources` 追溯回原始文件。
