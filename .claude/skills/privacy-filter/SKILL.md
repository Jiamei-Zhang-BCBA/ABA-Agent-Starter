**真名、全名、身份证、学校全称、公司名、门牌号、微信号** 这类 PII 只允许在**唯一一个文件**中出现：

| 文件 | 真名 / PII 是否允许 | 规则 |
|:---|:---|:---|
| `00-RawData/身份映射对照表-绝密.md` | ✅ **唯一合法出处** | 用于反向查询 |
| `00-RawData/脱敏存档/[代号-脱敏原始数据].md` | ❌ **绝对禁止** | 必须全部替换为代号 / 角色代称 |
| `04-Supervision/系统变更日志.md` | ❌ **绝对禁止** | 只写代号、脱敏动作摘要 |
| 其它任何文件 | ❌ **绝对禁止** | 含初访/核心/评估/IEP/FBA/家书/喜报/日志/实操单 |

**极易踩坑的半脱敏写法 — 严禁使用**：
- ❌ `Client-A-小磊 母亲（王丽，38岁，某银行部门总监）` — 把代号和真名**并列**出现 = 映射被泄漏，等同于未脱敏
- ❌ 变更日志里写 `真实身份：张子磊` / `个案真实姓名：xxx`
- ❌ 家庭简历段落里对父母各写一遍真名 + 职位（即使上方已给代号）
- ❌ 日记/备忘里"XX 老师就是王丽的妹妹"这种暗桩

**正确写法**：
- ✅ `Client-A-小磊 母亲（38岁，某股份制银行部门管理层）` — 只角色代称 + 去名化特征
- ✅ 变更日志：`个案代号：Client-A-小磊；数据来源：初访家长访谈笔录；脱敏操作完成。` — 完全不提真名
- ✅ 人物关系统一使用"父亲/母亲/爷爷/奶奶/保姆/班主任"角色代称，辅以"家庭代号"（如"磊磊爸"）

# 🛑 真名变体全捕获 (P1 硬规则 — BUG #23 对应)

**问题背景**：家庭成员访谈中互相使用的"姓氏 + 职业敬称 / 亲属称谓"在 v4
测试暴露脱敏缺口——真名 `李建国 / 陈婉仪` 被正确替换，但文中的 `李老师 / 陈老师 / 李教授 / 陈教授` 未被识别为同一人的代称，8 处残留在脱敏存档。

**识别算法**（每个真名 X 都必须做以下扩展检索）：

当身份映射表中录入真名 `姓 + 名` 后，自动识别并替换该人的**所有变体**：

| 变体类型 | 示例（以 "李建国" 为例） | 处理 |
|:---|:---|:---|
| 姓氏 + 职业敬称 | 李老师、李教授、李医生、李律师、李工、李经理、李总、李书记、李主任、李先生 | 全部替换为 `[Client-代号 父亲]` 或统一角色代称 |
| 姓氏 + 职称缩写 | 李博（博士）、李副（副教授）、李总工、李局 | 同上 |
| 姓氏 + 女士 / 先生 | 李先生、陈女士、王太太 | 同上 |
| 昵称变体 | 建国、国国、小李（对外人自称） | 需配合上下文判断是否代称同一人 |
| 复合关系暗桩 | "李教授的夫人就是陈老师"、"磊磊爸就是张强"、"她家老李" | **此类暗桩必须整句删除或彻底改写**，绝不能保留"暗示链" |

**扩展检索模板**（对每个身份映射表里的真名 `Name = 姓 + 名`，脱敏引擎必须做）：

```
For name in known_names:
    surname = name[0]  # 单姓；若复姓特殊处理（欧阳/司马等）
    variants = [
        surname + "老师", surname + "教授", surname + "医生",
        surname + "律师", surname + "总", surname + "博",
        surname + "经理", surname + "主任", surname + "书记",
        surname + "处长", surname + "先生", surname + "女士",
        surname + "太太", surname + "工程师", surname + "局长",
        name[1:],  # 去姓的名字
        "小" + surname, "老" + surname,
    ]
    for v in variants:
        if v in text:
            flag_for_review(v, context=neighbor_100_chars)
            text = text.replace(v, canonical_role_reference)
```

**边界情况**：
- **非 PII 姓氏撞车**：如果文中出现"李医生"但这是**另一个**真人（非家庭成员），也应脱敏为"某医院医生"
- **保留职业无姓**：`医生`、`老师`（无姓前缀）单独出现时**不必**脱敏（职业本身非 PII）

**本规则适用场景**：第二步"执行多准则脱敏"时，必须对身份映射表里**每一个真名**执行一次"变体扩展扫描"。

# 🚫 元注释禁令 (P1 硬规则 — BUG #25 对应，v5 新发现)

**问题背景**：v5 测试发现，privacy-filter 自己的 **meta-commentary**（FILE marker 之前的解释段）与变更日志中，AI 倾向于用"已处理/已替换"的证明性措辞**重新引用原真名**作为合规报告：

错误示例（v5 实际产生，全部属于**落盘泄漏**）：
- ❌ "已识别并替换以下真名及其变体：王小满(儿童) → Client-A-小满"
- ❌ "刘雅琴(母亲) → Client-A-小满 母亲"
- ❌ "BUG#23 特殊捕获（李老师 → 早教中心老师）：8 处全替换"
- ❌ "原始资料中的父母姓名(李建国/陈婉仪)已处理"（这是 BUG #24 同型，在 assessment 出现）

**本质**：即使意图是"展示合规证据"，只要真名被**写到任何会落盘的文本**（包括 meta-commentary、变更日志、自检清单），就违反了"真名分层管控"。

**硬规则**：

1. **FILE marker 之前的 preamble（meta-commentary）不得出现任何真名**。若需说明脱敏过程，只能说"已替换儿童真名 / 已替换母亲真名 / 已替换姓氏+职业敬称变体 N 处"，不得给出真名或原→新对照。

2. **变更日志**（`04-Supervision/系统变更日志.md`）里禁止出现真名作为"已替换证明"。正确示例：
   - ✅ "BUG#23 姓氏+职业敬称变体已全量处理（X 处命中）"
   - ✅ "新增 1 儿童 + 3 家庭成员映射"
   - ❌ "（李老师 → 早教中心老师）：8 处全替换"

3. **自检清单的陈述性描述**可以说"本轮捕获了父职敬称 + 母职敬称"，不得引用真实姓氏或名字片段。

4. **唯一合法"真名 + 代号"并列出处**仍然只有 `00-RawData/身份映射对照表-绝密.md`。任何其他落盘文本**即使是 AI 自己的解释/报告**都遵循"真名分层管控"铁律。

# 🏷️ 商标 / 品牌 / 特定商品脱敏扩展 (P2 硬规则 — BUG #27 对应，v5 新发现)

**问题背景**：v5 A-小满个案中，强化物"某特定品牌棉花糖"（原名"雪香"）在第一轮 privacy-filter 已正确替换为"某品牌"，但后续 session-reviewer / reinforcer / milestone-report 等 skill 的 user form 或 upload body 为了"Teacher A 能买对品牌"又引入真品牌名，导致 5 个下游档案泄漏真品牌。

**识别**：下列所有属于"商标 / 商品 / 机构专有名词"范畴：

| 类型 | 示例 | 处理 |
|:---|:---|:---|
| 食品 / 零食品牌 | 雪香棉花糖、乐事薯片、伊利牛奶 | 替换为"某品牌+品类"（"某品牌棉花糖"）|
| 玩具 / 电子品牌 | 乐高、小米平板、iPad | 替换为"某品牌+品类"|
| 机构全名 | 北京儿童医院、北大附小、回音壁学校 | 替换为"某儿童专科医院/某公立小学/某融合学校"|
| 医药品牌 | 舍曲林 (Zoloft)、阿立哌唑 | 保留**通用名或类别**（"SSRI 类抗抑郁药" / "抗精神病药"），不保留商品名 |
| 学习机构名 | "未来之星蒙氏早教" | 替换为"某早教机构" |

**追加到映射表**：当脱敏品牌 / 商标时，也必须在身份映射表新增一行：

```
| 真实品牌名 | 系统代号 (Aliases) | 录入日期 | 备注 |
|:---|:---|:---|:---|
| 雪香原味棉花糖 | 某品牌棉花糖 | 2026-04-19 | 强化物品牌 — 下游 skill 若需实操购买信息，请查本表 |
```

**下游 skill 的责任**：

- `session-reviewer` / `reinforcer-tracker` / `program-slicer` / 其它使用品牌信息的 skill，**必须**从身份映射表反查真品牌名（如需），但档案正文一律保留"某品牌"代称。
- Teacher A 的**实操单**（`03-Staff/教师-X/实操单-*.md`）允许使用真品牌名（教学需要），但应在文件 frontmatter 标注 `品牌信息敏感度: 🟠`。
- **正式档案**（核心档案 / IEP / 里程碑报告 / 家书 / 喜报）：一律保持"某品牌"代称。

**本规则适用场景**：第二步"执行多准则脱敏"与第三步"更新绝密映射表"时，品牌 / 商标视为与姓名同级的 PII 参与映射与替换。

# 执行步骤与多重文件操作
请你必须严格按照以下顺序，在本地执行**脱敏与绝密归档**操作：

**第一步：绝密对照表检索 (Read & Match)**
1. **指令**：读取 `00-RawData/身份映射对照表-绝密.md`。
2. **逻辑判定**：
   - 检查输入文本中的真实姓名是否已存在。
   - **已有案例**：沿用表中对应的系统代号（如 `Client-A-兜兜`）。
   - **新增案例**：根据表中现有数量，自动分配下一个顺延代号（如 `Client-C-小明`）。

**第二步：执行多准则脱敏 (De-identify)**
1. **强制替换**：将所有真实姓名替换为分配的代号。
2. **模糊处理**：将具体地址、学校全称、父母全名等替换为描述性词汇（如"某公立小学"、"Client-A 妈妈"）。
3. **保留专业度**：严禁修改任何关于行为频次、前因后果（ABC）、干预时长等专业数据。

**第三步：更新绝密映射表 (Append - 仅限新个案)**
1. **操作指令**：如果是新出现的孩子，请将对应关系追加到映射表末尾。

   - 追加格式：`| 真实姓名 | 系统代号 (Aliases) | 录入日期 |`

**第四步：存入脱敏存档库 (Write)**
1. **预览确认**：将脱敏后全文输出给督导，确认无遗漏后执行写入。
2. **操作指令**：新建文件写入净化后的文本。
