### 现场观察

```

```

（即便系统已进入 retry 1/2，新 retry 的 subprocess 依然未能在 600s 内完成 — 可能进入死循环或 silent fail）

### 根因推测（按可能性排序）

1. **fba skill 的 prompt 内部要求"遍历全部 session 日志 + 核心档案 + intake + assessment"** — A-小舟 已完成 Step 1-10，vault 有 45+ 个文件，fba skill 尝试读太多导致 context 设置阶段就超时
2. **retry 后 timeout 计数未重置** — 可能 600s 是"跨 retry 累计"而非"单次 attempt"
3. **并行提交时 subprocess 资源竞争** — 4 个 Claude CLI subprocess 同时跑，CPU/内存被其他 3 个 auto job 抢占，FBA 的 subprocess 实际只有部分时间片

### 影响

- FBA 是 ABA 临床的核心技能（问题行为功能分析）
- 没有 FBA 就无法生成 BIP（行为干预方案）
- BIP 缺失 → 对小舟的拍头自伤问题**没有可执行的干预档案**
- BCBA 无法据此做临床决策 → **完整生命周期闭环断裂 92%**（12/13）

### 建议修复（按优先级）

**P0 - 立即**:
1. **检查 local_worker 的 retry 机制**：确认 timeout 后新 subprocess 是否真正启动（观察到 tokens 一直 0 很可疑）
2. **延长 fba 的 timeout**：`JOB_TIMEOUT_SECONDS=900`（至少 1.5x）或 per-skill 配置

**P1 - 短期**:
3. **fba skill prompt 压缩**：把"读全部日志"改为"读最近 3 条日志 + ABC 附件"
4. **expert tier 独立 queue**：避免 auto 并发把 expert 挤出去

**P2 - 中期**:
5. **fba 拆两步**：ABC-extractor (auto, 3min) + hypothesis-writer (expert, 5min)
6. **加 job 死锁检测**：queued 超过 `2 * job_timeout_seconds` 直接 fail

### 可能的快速验证

**单跑测试**：不并发其他 job，单独提交 fba，看能否在 600s 内完成。如果单跑通过 → 并发是主因；如果单跑也超时 → skill prompt 本身问题。

## 🟢 观察 1: privacy_filter 前端字段名不匹配后端

- **模块**: 模块 3-1
- **严重度**: 🟢 文档/约定问题（不影响走 UI 流程的用户，影响 curl/自动化）

### 现象

通过 API 直调 `POST /api/v1/jobs` 时，如果 multipart 字段名使用 form_schema 里定义的 `raw_file`（privacy_filter 专属），后端 `files: list[UploadFile] = File(...)` 收不到文件 → AI 没拿到内容 → 返回"请上传访谈记录"。

### 根因

`app/routers/jobs.py` 强制约定**所有** skill 的 upload 字段名都是 `files`，不管 form_schema 里写的是什么名字。form_schema 的 `name` 只控制前端 UI 显示的 label 和 `formData[name]` 绑定。

前端 `JobFormModal` 其实已经用了 `fd.append("files", file)`（正确），所以 UI 用户不会踩坑；但 API 文档或自动化测试的编写者会困惑。

### 建议

在 `api/app/routers/jobs.py` 或 API 文档里明确备注：**multipart 文件字段名永远是 `files`，form_schema 里 `type=file` 的 field name 只影响前端展示**。
