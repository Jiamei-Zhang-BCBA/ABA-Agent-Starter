## 🔴 BUG #18: API 直调时 client_id 必须同时出现在 top-level form 字段与 form_data JSON，否则 vault 不写盘（但 AI 已收 token 并返回 review）

> **状态**: 🟡 **OPEN**，可绕过（API 调用方注意），但应改为 skill_executor 自动补全
> **严重度**: 🟡 P1（UI 用户不受影响，只影响 API 直调/自动化/curl 工作流；会静默浪费 expert tier 成本）

### 现象（模块：REST API）

通过 `POST /api/v1/jobs` 提交需要绑定 client 的 skill（profile_builder / assessment / plan_generator / fba / program_slicer / session_review / reinforcer / parent_update / milestone_report / transfer_protocol）时：

- ✅ **必须**在 multipart 顶层 Form 字段放 `client_id=<uuid>` （决定 `job.client_id` 入库字段）
- ✅ **同时**在 `form_data` JSON 里也放 `client_id`（通过 form_validator 必填校验；否则返 `400 必填字段缺失`）

**只放 form_data 不放顶层**：
- `POST /jobs` 返回 201，`job.client_id = null`（API 响应 body 里可见）
- Skill 正常跑（AI 看到 form_data 里的 client_id 字符串），返 pending_review，output_content 含完整 `<!-- FILE: -->` markers
- approve 成功（200），但 `review_service.approve_review` 第 138-154 行逻辑：
  1. `client_code = form_data.child_alias` → 为空（本 skill form_schema 没这字段）
  2. 回退 `client_code = job.client_id.lookup(Client).code_name` → `job.client_id=null` 走不通
  3. 最终 `client_code=""`，logger.warning + 跳过 `write_output_to_vault()`
  4. **vault 文件完全未被更新**，但 job 状态已 `delivered`
- 用户从 UI 或 vault 列表看档案，发现还是占位骨架（`63 字的 "⏳ 此文件将在 xxx 技能执行后自动填充"`）

### 复现证据

**第一次提交**（错误：client_id 只放 form_data）：
```python
r = requests.post(BASE+'/jobs',
    data={'feature_id':'assessment',
          'form_data': json.dumps({'client_id': CID, 'tool_name':'VB-MAPP'})},
    files={'files': (...)})
# 201，job.client_id=None；AI output_content=7862 chars with 3 FILE markers
# approve 200；vault 读回 能力评估.md 仍然 63 chars 骨架
```

**第二次提交**（正确：两处都放）：
```python
r = requests.post(BASE+'/jobs',
    data={'feature_id':'assessment',
          'client_id': CID,  # <-- 顶层
          'form_data': json.dumps({'client_id': CID, 'tool_name':'VB-MAPP'})},  # <-- form_data 也有
    files={'files': (...)})
# 201，job.client_id=<uuid>；AI out=10527 chars
# approve 200；vault 读回 能力评估.md = 3785 chars 含 14 独特信号 ✅
```

### 根因

`api/app/routers/jobs.py:50` 声明 `client_id: str | None = Form(None)` 顶层字段；jobs.py:121-130 只用顶层字段写入 `Job.client_id` DB 列。form_data 里的 client_id 仅供 form_validator 校验 + 透传给 AI prompt，**不会回填到 `Job.client_id`**。

`review_service.approve_review` 在决定 `client_code` 时优先用 form_data.child_alias（只有 intake/privacy_filter 有），再回退 `Job.client_id` → `Client.code_name`。两个通路都断裂时静默跳过 vault write。

### 影响 / 成本

- **静默数据丢失**：用户以为 skill 成功了，实际档案没更新
- **expert tier 浪费**：每次失败消耗 ~30k input + 10k output tokens（约 $0.15/次）
- **自动化脚本 / curl 调用方极易踩坑**：前端 UI 因为有 JobFormModal 的 client_id 组件同时填两处，用户看不到这个坑
- **上一轮测试（A-小舟）没踩**：因为走的是浏览器 + JobFormModal，已自动双写

### 建议修复

**P0 - 立即**：
1. `jobs.py:121-130` POST 处：如果顶层 `client_id=None` 但 `form_data.client_id` 存在，自动同步到 `Job.client_id`（写入 DB）
   ```python
   if not parsed_client_id:
       fd = json.loads(form_data) if form_data else {}
       if fd.get('client_id'):
           parsed_client_id = fd['client_id']
           await _check_client_access(db, user, parsed_client_id)
   ```

**P1 - 短期**：
2. `review_service.approve_review:154` 现在的 "No client_code for job {id}, skipping vault write" warning 应升级为 **ERROR**，并把失败原因写入 review.comments 让调用方能查到
3. API 文档（OpenAPI / docs/plans/）里明确说"顶层 client_id 必填（决定 DB 绑定），form_data.client_id 决定 AI prompt 注入"

**P2 - 中期**：
4. 统一方案：skill 对 client_id 的依赖应由 feature_registry 声明 `requires_client=True`，后端自动校验两处一致性（不一致则 400）

### 相关文件

- `api/app/routers/jobs.py:50, 121-130`
- `api/app/services/review_service.py:100-156`
- `api/app/services/vault_service.py:445-452`（当 client_code 缺失时的保护分支）

## 🟡 观察 1: profile_builder 无 code_name 提示时 AI 会"要求补代号"（影响整体成本）

### 现象

当 profile_builder 的 form_data.additional_context 里**未显式写出代号** `A-小航` 时：
- AI 收到 "client_id=<uuid>" 但不知道对应代号是什么
- AI 不敢猜，直接输出 `请提供：代号字母、系统内使用的昵称，提供后立即生成`
- 全文 172 字，0 个 FILE marker，但算作 `delivered` 状态，浪费了 25851 input token + 1047 output token（约 $0.1）

### 原因

`skill_executor.py` 构造 user_message 时应该：
- 已知 `client_id` → 查 `Client.code_name` → 在 prompt 里写入 `档案代号：A-小航（锁定绑定，不要自行推导）`
- 目前**只透传了 form_data JSON**，AI 看不到 code_name 字面值

### 绕过方式（已验证）

在 additional_context 里显式写 `**儿童代号：A-小航，昵称小航**` 作为第一段，AI 就能正确工作。

### 建议

skill_executor 的 `_build_user_message`（见 MEMORY.md 里提到的"档案代号绑定"硬规则）应该对**所有** `requires_client=True` 的 skill 都注入 `档案代号：{code_name}`，不只是对 upload 有冲突风险的 skill。

## 🔴 BUG #19: vault_service 的 FILE marker 解析器不支持 `| EDIT:章节名` 语法

> **状态**: 🟡 **OPEN**，必须修复（FBA / assessment / 其他需要编辑既有章节的 skill 都受影响）
> **严重度**: 🔴 **P0**（数据污染 + 章节未合并 = 临床数据分散）

### 现象

FBA skill 输出：
```
<!-- FILE: 01-Clients/Client-A-小航/Client-A-小航-核心档案.md | EDIT:🚨 历史问题行为备忘 -->
### 🚨 历史问题行为备忘（基于 2026-04-22 FBA 正式更新）
... 1922 字 FBA 结果 ...
```

**期望**：在核心档案的 `### 🚨 历史问题行为备忘` 章节**替换**为新内容。

**实际**：整段 `Client-A-小航-核心档案.md | EDIT:🚨 历史问题行为备忘` 被当路径写了脏文件：
```
01-Clients/Client-A-小航/Client-A-小航-核心档案.md | EDIT:🚨 历史问题行为备忘  (1922 字)
```

核心档案本体**未被编辑**，FBA 章节更新游离在孤立文件里。

### 根因

`api/app/services/vault_service.py:372` regex：
```python
r'<!--\s*FILE:\s*(.+?)(?:\s*\|\s*(APPEND))?\s*-->'
```
只识别 `| APPEND`，`| EDIT:xxx` 整段被 `(.+?)` 吃进 group(1) 当路径。

### 影响

- 档案章节游离：核心档案里 FBA 更新看不到
- vault 目录污染：01-Clients 下出现非法文件名（管道符、冒号）
- Obsidian `[[]]` 双链无法指向
- 所有用 `| EDIT:` 的 skill 受影响（FBA / assessment 某些情况 / profile_builder 部分章节）

### 建议修复（P0）

```python
r'<!--\s*FILE:\s*([^|]+?)(?:\s*\|\s*(APPEND|EDIT:([^>]+?)|MERGE))?\s*-->'

# 分支：
# 无修饰 → write_file（覆盖）
# APPEND → existing + "\n" + new
# EDIT:章节名 → 读原文件 → 用 _config.md 章节锚点定位 → 替换该章节
# MERGE → 智能合并（短期先降级为 APPEND）
```

### 相关文件

- `api/app/services/vault_service.py:367-435`
- `.claude/skills/fba-analyzer/SKILL.md`（输出 EDIT marker 的源头）
- `.claude/skills/_config.md:65-75`（章节锚点定义）
